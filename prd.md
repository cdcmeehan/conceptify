**Version:** 0.1 (draft) В· **Date:** 2026-07-01 В· **Owner:** Chris Meehan В· **Status:** Ready for task breakdown

## 1. Overview

Conceptify is a personal macOS desktop app for deeply understanding concepts: codebases, code flows, unfamiliar languages/frameworks, and general technical questions. AI coding agents (Claude Code first) generate rich, visual, self-contained HTML "explanation artifacts" and publish them into Conceptify, where they are rendered in a beautiful reading experience. The user can then interrogate the explanation вҖ” highlight text or diagram elements, leave comments, and trigger a background agent to answer follow-ups in a sidebar and, when needed, update the artifact live.

**One-line pitch:** a personal "grokking workbench" вҖ” agents explain, Conceptify renders, you interrogate until it clicks.

### 1.1 Why not just chat / markdown files / a browser?

- Chat transcripts are ephemeral and text-first; explanations here are **durable, visual documents** organized by project.
- Artifacts are **portable HTML** вҖ” they open in any browser вҖ” but Conceptify adds the interactive layer (comments, follow-ups, live updates) that a static file can't have.
- The app is the **hub between agent sessions**: a Claude Code session ends, but the artifact, its comments, and the ability to spin up a new headless agent to refine it persist.

### 1.2 Primary user

Just Chris. Single-user, single-machine (macOS), personal tooling. No accounts, no sync, no multi-tenancy. This dramatically simplifies security, distribution, and scope decisions throughout this document.

## 2. Goals & Non-Goals

### Goals

1. **G1 вҖ” Fast, native-feeling app.** Tauri v2, cold start well under 1s, small footprint. No Electron.
2. **G2 вҖ” Beautiful explanations.** A baked-in design system (modern serif headings, readable sans body, styled diagrams, tasteful animation) that makes artifacts a delight to read.
3. **G3 вҖ” Portable artifacts.** Every answer is a self-contained `.html` file that renders correctly in Conceptify's webview *and* standalone in a browser.
4. **G4 вҖ” Agent-native workflow.** A Claude Code skill + `conceptify` CLI make "use conceptify to explain this" a one-line request from any codebase.
5. **G5 вҖ” The interrogation loop.** Highlight/comment вҶ’ sidebar answers вҶ’ optional live artifact update. This is the crown-jewel differentiator.
6. **G6 вҖ” Agent-agnostic by design.** Built and tested against Claude Code, but agent invocation is a configurable adapter so Codex (or anything) can slot in later.

### Non-Goals (explicitly out of scope)

- Multi-user, auth, cloud sync, mobile, Windows/Linux (Tauri keeps the door open; we do nothing to support them).
- Being a general chat client. Threads are explanation-centric, not conversation-centric.
- Hardened security against a malicious agent. Artifacts are AI-generated by tools the user already trusts with shell access; sandboxing is about containment and hygiene, not adversarial defense (see В§9).
- Video export (Remotion etc.) in Phase 1 вҖ” see В§11 Phase 2.

## 3. Core Use Cases

- **UC1 вҖ” "Explain this" from a coding session.** In Claude Code inside any repo: *"use conceptify to explain how auth middleware works in this codebase."* The skill ensures the app is running, ensures a project exists for the repo (mapped by root path, auto-named from dir name), creates a thread, generates a visual artifact, saves it, and the app opens/focuses that thread.
- **UC2 вҖ” "What is this syntax?"** Quick unfamiliar-language/framework questions (*"what does this Rust lifetime annotation mean?"*) producing a compact tutorial artifact with annotated code.
- **UC3 вҖ” Whole-codebase orientation.** Just cloned a repo; ask for a guided architecture tour: layered diagrams, key flows, entry points, directory map.
- **UC4 вҖ” Interrogate the artifact.** While reading: highlight a sentence вҶ’ "I don't get why the token is refreshed here" вҶ’ comment saved. Click a diagram node вҶ’ comment on that node. Hit **Ask follow-ups** вҶ’ background headless agent answers each comment in the sidebar. If an answer implies the artifact should be clearer, choose **Apply to artifact** вҶ’ agent edits the HTML, app refreshes live, comment marked resolved-with-update.
- **UC5 вҖ” Ask directly from Conceptify.** Open a project in the app, type a question, agent runs headless from the project's mapped directory, artifact appears when done (with streaming progress indication).
- **UC6 вҖ” Create a project in-app.** Pick a directory, or let Conceptify auto-create `~/Documents/conceptify/projects/<name>/` for non-codebase topics (e.g. "distributed systems", "music theory").

## 4. Domain Model

| Entity | Description | Key fields |
|---|---|---|
| **Project** | A workspace mapped 1:1 to a root directory (a repo, or an auto-created topic folder). | id, name, root_path (unique), created_at, archived |
| **Thread** | One question/topic within a project. Owns exactly one artifact (versioned) plus its comment history. | id, project_id, title, initial_question, status (`generating` \| `ready` \| `updating` \| `error`), created_at, updated_at |
| **Artifact** | The self-contained HTML answer. Versioned: every agent update creates a new version; prior versions retained. | id, thread_id, version, file_path, created_at, created_by (`initial` \| `follow_up`) |
| **Comment** | A user annotation anchored to a region of the artifact, optionally with agent resolution. | id, thread_id, artifact_version, anchor (JSON, see В§7.4), body, status (`open` \| `answered` \| `applied`), answer_html, created_at, resolved_at |
| **FollowUpRun** | One execution of a background agent handling a batch of comments (or a direct follow-up question). | id, thread_id, agent, model, mode (`answer` \| `apply`), status, log_path, started_at, finished_at |
| **Settings** | Global app configuration. | default agent adapter, follow-up model, artifact-update model, theme, editor, etc. |

**Direct follow-up questions** (typed at the bottom of a thread, not anchored to a selection) are modeled as Comments with a null anchor вҖ” they flow through the identical sidebar/resolution machinery.

### 5.1 Rust core (Tauri v2)

- **HTTP API:** axum 0.8 server spawned in the Tauri `setup` hook via `tauri::async_runtime::spawn`, bound to `127.0.0.1:4477`. The `AppHandle` is shared into axum state so HTTP handlers can emit Tauri events (`artifact-updated`, `comment-resolved`, `thread-created`) that drive live webview updates.
  - **Auth:** random bearer token generated on first run, written mode `0600` to `~/Library/Application Support/conceptify/token`; the CLI reads it. All endpoints except `GET /health` require it. (Any local process or browser page can reach localhost ports вҖ” the token prevents drive-by access.)
  - **Port conflicts:** on `AddrInUse`, probe `/health`; if another Conceptify answers, defer to it (single-instance guard should prevent this); otherwise fall back through 4477вҖ“4487 and write the bound port to `~/Library/Application Support/conceptify/port` for CLI discovery.
- **Database:** rusqlite (bundled SQLite, WAL mode) held in Tauri managed state, shared by both `#[tauri::command]` handlers and axum handlers. **Not** `tauri-plugin-sql` вҖ” it has no first-class Rust-side query API, and our server code needs direct DB access. Migrations via `rusqlite_migration` or a `user_version` scheme.
- **Agent spawner:** `tokio::process::Command` (not tauri-plugin-shell вҖ” that plugin's value is frontend-initiated execution, which we don't allow). `kill_on_drop(true)`, children tracked in managed state, stdout/stderr streamed line-by-line to the frontend via Tauri events. GUI apps on macOS get a minimal `PATH` вҖ” resolve the agent binary via a login-shell lookup (`zsh -lc 'which claude'`) once and cache it, or let settings override the absolute path.
- **Lifecycle:** `tauri-plugin-single-instance` as a duplicate-launch guard; window close **hides** instead of quitting (`CloseRequested` + `prevent_close`) so the API stays up while no window is visible; explicit Quit from the app menu/tray. `tauri-plugin-window-state` for size/position persistence.

### 5.2 `conceptify` CLI

Thin, fast client for the local API. Distributed as part of this repo, installed onto `PATH` (symlink into `/usr/local/bin` or `~/.local/bin` via a `just install` / setup script). Implementation: a small Rust binary in the same Cargo workspace (shared types with the server) вҖ” avoids a Node dependency and starts instantly.

**Liveness/launch contract:** every command first probes `GET /health` (using the port file); on failure it runs `open -a Conceptify`, polls `/health` up to ~10s, then proceeds. This sidesteps macOS single-instance/argv-forwarding quirks entirely вҖ” "launch and focus" is just HTTP.

Commands (mirroring API endpoints, JSON output on stdout for agent consumption):

```
conceptify ensure-project --dir <path> [--name <name>]     вҶ’ {"projectId":вҖҰ, "created":bool}
conceptify create-thread  --project <id> --title <t> --question <q>  вҶ’ {"threadId":вҖҰ}
conceptify save-artifact  --thread <id> --file <path>      вҶ’ {"version":n}   (validates, stores, live-refreshes, focuses)
conceptify get-context    --thread <id>                    вҶ’ thread + question + artifact path + open comments (JSON)
conceptify list-comments  --thread <id> [--status open]    вҶ’ comments with anchors (JSON)
conceptify resolve-comment --id <id> --answer-file <path> [--applied]  вҶ’ marks answered/applied, updates sidebar live
conceptify open           --thread <id> | --project <id>   вҶ’ focus app on target
conceptify status                                          вҶ’ app/API health, version
```

### 5.3 Frontend shell (Preact + Tailwind v4)

- Scaffold: create-tauri-app (React template) swapped to Preact via `@preact/preset-vite` (the official Preact template was removed in create-tauri-app v3); Vite; Tailwind v4 via `@tailwindcss/vite`.
- Views: **Project list** (sidebar) вҶ’ **Thread list** per project вҶ’ **Thread view** (artifact viewer + comments sidebar + follow-up composer) вҶ’ **Settings**. Keyboard-first navigation where cheap (вҢҳK switcher is Phase 2).
- Live updates: subscribes to Tauri events emitted by the Rust core; no polling.

### 5.4 Artifact rendering & isolation

- Artifacts are served through a **custom `artifact://` URI scheme** (`register_asynchronous_uri_scheme_protocol`): the handler reads the HTML file from disk, injects the **bridge script** (see below), and responds with a restrictive per-response CSP (allow inline script/style; `connect-src 'none'` so artifact JS can never reach the localhost API; allow the pinned CDN hosts from В§8 Tier 2).
- The viewer embeds `<iframe src="artifact://localhost/<thread>/<version>" sandbox="allow-scripts">` вҖ” **no `allow-same-origin`**, so the artifact runs in an opaque origin: it cannot touch the app DOM, Tauri IPC, storage, or the API. App-shell CSP in `tauri.conf.json` stays strict and allows `frame-src artifact:`.
- **Bridge (postMessage protocol):** the injected script provides the in-artifact interaction layer:
  - text selection вҶ’ reports selection ranges/quotes to the shell for the "leave comment" popover;
  - click-to-comment on diagram elements (anything bearing `data-cfy-id`, see В§7.3);
  - rendering highlight decorations for existing comment anchors;
  - scroll-to-anchor when a sidebar comment is clicked.
  - The shell validates `event.source === iframe.contentWindow` on every message (opaque-origin frames report `origin: "null"`).
- Because injection happens server-side in the protocol handler, the `.html` file on disk stays pristine and portable вҖ” opened in a browser it's just a beautiful static document, no Conceptify residue.

### 5.5 Agent integration layer

Two distinct integration surfaces:

1. **Authoring (interactive):** the Claude Code **skill** (В§7.6) teaches an already-running agent session how to produce artifacts and publish them via the CLI. Conceptify is passive here вҖ” it just receives.
2. **Follow-ups & in-app asks (headless):** Conceptify itself spawns an agent process. This goes through the **agent adapter** вҖ” a settings-defined command template, e.g.:

```jsonc
{
  "adapters": {
    "claude": {
      "command": "claude",
      "args": ["-p", "{prompt}", "--model", "{model}", "--permission-mode", "acceptEdits", "--output-format", "stream-json"],
      "cwd": "{project_root}"
    }
    // future: "codex": { "command": "codex", "args": ["exec", вҖҰ] }
  },
  "defaultAdapter": "claude",
  "models": { "followUp": "claude-haiku-4-5", "artifactUpdate": "claude-sonnet-5", "inAppAsk": "claude-sonnet-5" }
}
```

Phase 1 ships and tests only the `claude` adapter, but all invocation flows go through this template so nothing is hardcoded. Per-purpose model config satisfies "don't burn a frontier model on small sidebar answers."

The headless agent gets its instructions from the same globally-installed skill files (so authoring rules live in exactly one place) plus a run-specific prompt assembled by Conceptify containing: thread question, artifact path, and the open comments with anchors (via `conceptify get-context`). It reports results back through the CLI (`resolve-comment`, `save-artifact`), which the app reflects live.

### 5.6 Storage

- **SQLite** (`~/Library/Application Support/conceptify/conceptify.db`): all metadata вҖ” projects, threads, comments, follow-up runs, settings.
- **Files on disk**: artifacts live **centralized**, keyed by project id, not inside the mapped project directory (resolved as **OQ2**, В§12 вҖ” decision recorded on bead `conceptify-36s.7`):

```
~/Documents/conceptify/
  artifacts/
    <project-id>/                 вҶҗ opaque, immutable; stable across project rename/re-map (FR-1.3)
      threads/
        <thread-slug>/
          artifact.v1.html
          artifact.v2.html        вҶҗ follow-up updates append versions
          artifact.html            вҶҗ atomic copy of the latest version (temp+rename, N4), for one-click "Open in browser"
          runs/<run-id>.log       вҶҗ headless agent transcripts, for debugging
  projects/
    <slugified-name>/              вҶҗ unchanged: root_path for auto-created (non-codebase) projects only вҖ” a working directory for agents, distinct from artifact storage above
```

`<project-id>` is the DB primary key, so moving, renaming, or temporarily losing a mapped repo directory (already anticipated by FR-1.3's re-map affordance) never touches artifact files вҖ” only the `root_path` mapping needs repair. The `artifact.html` alias is a real file (copy), not a symlink, written via the same atomic temp+rename discipline as versioned files, so a stale/dangling link is never possible.

**Rationale (decided during design):** metadata needs transactional integrity and cheap querying (comment status, thread lists) вҶ’ SQLite; artifacts need portability and hand-openability вҶ’ plain files, but centralized rather than embedded in the (git-managed, user-volatile) project directory вҖ” see OQ2 decision. The DB stores paths, never HTML blobs.

## 6. Design System & Reading Experience

The design system is what makes artifacts "a delight to read" вҖ” it matters more than any diagram library choice. It ships in two forms that must stay in sync: (a) a CSS scaffold **embedded into every artifact** by the skill (so artifacts are self-contained), and (b) the same tokens used by the app shell for visual continuity.

- **D1 вҖ” Typography:** modern serif for headings (candidates: *Newsreader*, *Fraunces*, *Source Serif 4*); highly readable sans for body (*Inter* or *system-ui* stack); mono for code (*JetBrains Mono* / *SF Mono* stack). Fonts referenced with robust system-stack fallbacks so artifacts never break offline; final picks during M2 design pass.
- **D2 вҖ” Tokens:** type scale, spacing rhythm, max line length (~68ch), and a full light/dark palette expressed as CSS variables. Dark mode via `prefers-color-scheme` so it works in-app and in browsers.
- **D3 вҖ” Content components** (classes the skill is instructed to use): callouts (insight / warning / definition), step sequences, key-term highlights, collapsible deep-dives (`<details>`), figure + caption blocks, comparison tables.
- **D4 вҖ” Code blocks:** pre-rendered **Shiki v4** at generation time вҖ” inline-styled spans, dual light/dark theme via CSS variables, zero runtime JS. Line highlighting and annotation callouts for walkthroughs.
- **D5 вҖ” Diagram styling:** diagrams consume the same CSS variables (D2 theme args / Mermaid `themeVariables` / hand-authored SVG using `currentColor` + `var()`), so every visual reads as one system. Optional hand-drawn aesthetic (D2 `--sketch` / Mermaid `handDrawn` look + a hand-writing display font) as a per-artifact flavor.
- **D6 вҖ” Motion:** CSS-first (staged reveals, `stroke-dashoffset` flow animation); compositor-friendly properties only (transform/opacity вҖ” WKWebView is 60fps-capped pre-macOS-26). Progressive enhancement: every artifact fully readable with JS disabled.

## 7. Feature Requirements

Numbered for task breakdown. Priority: **P1** = Phase 1 (all milestones), **P2** = Phase 2.

### 7.1 Projects вҖ” P1

- **FR-1.1** Ensure-project by directory: given a root path, return the existing project or create one, defaulting name to the directory name (deduped with suffix if taken). Path is canonicalized; one project per path.
- **FR-1.2** In-app creation: native directory picker, or "create a folder for me" вҶ’ `~/Documents/conceptify/projects/<slugified-name>/`.
- **FR-1.3** Project list UI with thread counts and last-activity; rename; archive (hide, don't delete). Handle mapped directories that no longer exist (badge + re-map affordance).

### 7.2 Threads вҖ” P1

- **FR-2.1** Create via API/CLI (agent flow) or in-app (ask flow). Fields per В§4; slug generated from title for the artifact folder.
- **FR-2.2** Thread list per project, sorted by last activity, showing status chips (`generating`/`ready`/`updating`/`error`) and open-comment counts.
- **FR-2.3** Thread view = artifact viewer + comments sidebar (В§7.4) + follow-up composer. While `generating` with no artifact yet, show streamed agent progress (В§7.7).
- **FR-2.4** Artifact version history: switch between versions; visual diff is P2 (nice-to-have, not required).
- **FR-2.5** "Open in browser" button (opens `artifact.html` with the system default browser) вҖ” the portability guarantee, permanently one click away.

### 7.3 Artifact specification вҖ” P1

The contract between generating agents and the app. Also published inside the skill.

- **FR-3.1** An artifact is a **single self-contained `.html` file**: inline CSS (design-system scaffold + artifact-specific), inline SVG diagrams, inline JS only when needed. No local file references. Network references only from the Tier-2 pinned-CDN allowlist (В§8), always with graceful degradation.
- **FR-3.2** Must render correctly in both WKWebView (Safari-compatible CSS/JS вҖ” no Chromium-only features) and standalone browsers.
- **FR-3.3** **Anchorability:** every section heading and every meaningful diagram element (nodes, edges, groups) carries a stable `data-cfy-id` attribute. The skill mandates this; `save-artifact` validation warns when coverage is thin. This is what makes diagram-level comments and robust anchor re-attachment possible.
- **FR-3.4** Diagram sources (D2/Mermaid/DOT text) embedded as HTML comments adjacent to their rendered SVG, so follow-up agents can regenerate rather than hand-edit SVG.
- **FR-3.5** Metadata block: `<meta name="cfy:question">`, `cfy:version`, `cfy:generated-by` in `<head>`.
- **FR-3.6** `save-artifact` validation: parseable HTML, size sanity cap (warn > ~5 MB), no `<script src>` outside the allowlist, `data-cfy-id` presence. Warnings surface to the agent (stderr/JSON) rather than hard-failing, except for allowlist violations.

### 7.4 Comments & the interrogation loop вҖ” P1 (crown jewel)

- **FR-4.1** **Text comments:** select text in the artifact вҶ’ popover ("Add comment") вҶ’ comment saved with anchor.
- **FR-4.2** **Diagram comments:** click any `data-cfy-id`-bearing element вҶ’ hover affordance + popover вҶ’ comment anchored to that element id.
- **FR-4.3** **Direct follow-ups:** free-text question at the bottom of the thread вҖ” a Comment with null anchor.
- **FR-4.4** **Anchoring model** (stored as JSON on the comment): primary = `data-cfy-id` + character offsets within that element; fallback = text-quote with prefix/suffix (W3C Web Annotation style) for re-attachment after artifact updates. If re-anchoring fails post-update, the comment is flagged "reference moved" in the sidebar вҖ” never silently dropped.
- **FR-4.5** **Sidebar:** all comments for the thread, filterable (open / answered / applied), each showing anchor excerpt, body, and вҖ” once resolved вҖ” the agent's answer rendered from HTML/markdown. Clicking scrolls the artifact to the anchor; open anchors show subtle persistent highlights in the artifact.
- **FR-4.6** **Ask follow-ups (batch, sidebar-first):** button gathers all open comments вҶ’ spawns one headless agent run (adapter + `models.followUp`) вҶ’ agent answers **each comment individually in the sidebar** via `resolve-comment`. **No artifact modification in this mode.** Sidebar updates live per-comment as answers land.
- **FR-4.7** **Apply to artifact (explicit second step):** per answered comment вҖ” or batch вҖ” an "Apply to artifact" action spawns a run in `apply` mode (`models.artifactUpdate`): agent edits/regenerates the relevant artifact portions (using embedded diagram sources where applicable), saves a **new version** via `save-artifact`, viewer refreshes live, comments transition to `applied`. Simple Q&A never needs this; it exists for "make this clearer" feedback.
- **FR-4.8** Run status UI: sidebar shows the in-flight run (spinner, per-comment progress, cancel button вҶ’ kills the process). Failures surface the log tail; `runs/<id>.log` retained.
- **FR-4.9** Concurrency guard: one follow-up run per thread at a time.

### 7.5 Ask from Conceptify вҖ” P1

## 8. Visual & Diagram Strategy (tiered)

Research conclusion (July 2026): **default to generation-time rendering with everything inlined; escalate to runtime libraries only when interactivity demands it.** The generating agent has a full toolchain вҖ” use it at authoring time, not in the reader's browser.

- **Tier 0 вҖ” always:** semantic HTML/CSS, design-system scaffold, CSS animation. Readable with JS off.
- **Tier 1 вҖ” DEFAULT for all diagrams вҖ” inline SVG, produced two ways:**
  - *Structured diagrams* (flow, architecture, sequence, state): agent writes **D2** (preferred: single Go binary, `--layout=elk`, optional `--sketch`, best-in-class output) or **Graphviz DOT** (pure graphs; LLMs write DOT extremely well) or Mermaid вҖ” **renders to SVG at generation time and inlines it**, DSL source embedded in an adjacent HTML comment (FR-3.4). Fully self-contained, offline, CSP-clean, identical in WKWebView and browser.
  - *Small bespoke visuals* (вүӨ ~10 elements: concept sketches, annotated pipelines, data-flow animations): agent hand-authors inline SVG + CSS/SMIL animation, gated by the self-review loop (FR-6.3). SMIL is fine in WebKit (main-thread вҖ” keep it light).
- **Tier 2 вҖ” pinned CDN allowlist (opt-in escalation, mirrored in the artifact CSP):** `mermaid@11` + ELK layout (only for *interactive* diagrams; always Neo/handDrawn look + themeVariables, never defaults), `motion@12` (2.5 kB mini) or `animejs@4` for JS-sequenced explainers, `gsap@3` for heavy timelines (fully free since 2025), `d3@7` for custom interactive viz, `katex@0.17` for math, `markmap` for interactive mind maps. Requirement: graceful offline degradation (diagram source visible as styled `<pre>` fallback).
- **Tier 3 вҖ” P2:** "scene-stepped artifacts" вҖ” sections driven by a GSAP/Motion timeline with play/scrub controls: ~80% of explainer-video value, zero new toolchain. Real video export (Remotion вҖ” active, free at personal scale) only if ever genuinely needed. Motion Canvas is abandoned upstream вҖ” avoid.
- **Code highlighting:** pre-rendered **Shiki v4**, always (D4). Fallback if node unavailable: highlight.js CDN.
- **P2 option:** vendor local copies of Tier-2 libs served via the artifact protocol with URL rewrite, for fully-offline in-app viewing.

## 9. Security Model (right-sized)

Threat model: single-user machine; artifact authors are agents the user already grants shell access. Sandboxing is containment and hygiene, not adversarial defense вҖ” a WebKit 0-day is out of scope.

- **S1** API on `127.0.0.1` only + bearer token (0600 file) вҖ” blocks other local processes and browser-page localhost probing (DNS-rebinding class).
- **S2** Artifacts in opaque-origin sandboxed iframes (no `allow-same-origin`), served cross-scheme (`artifact://`) with per-response CSP; `connect-src` restricted so artifact JS can't call the API; no Tauri IPC exposure (capabilities scoped to the main window; `withGlobalTauri` off).
- **S3** Frontend never executes shell commands; all process spawning is Rust-side through the adapter template.
- **S4** `save-artifact` allowlist enforcement (FR-3.6) keeps artifacts honest about external references.
- **S5** Distribution: local `tauri build`, ad-hoc signing, no notarization/Apple Developer account (personal machine only).

## 10. Non-Functional Requirements

- **N1 Startup:** cold launch to interactive < 1s on Apple Silicon (Tauri baseline: ~0.5s; benchmarks: 3вҖ“10 MB bundle, ~30вҖ“40 MB idle RAM vs Electron's 80вҖ“150 MB / 100вҖ“300 MB).
- **N2 Latency:** save-artifact вҶ’ visible refresh < 500ms; comment interactions instant (< 100ms perceived).
- **N3 CLI overhead:** < 50ms per command against a running app (excluding launch path).
- **N4 Resilience:** DB is WAL-mode SQLite; artifact writes atomic (temp + rename); a crashed follow-up run must never corrupt thread state (status machine + run log).
- **N5 Scale envelope:** designed comfortable at ~50 projects / ~1,000 threads; no pagination heroics needed.
- **N6 Artifact longevity:** Tier-0/1 artifacts must render correctly with zero network, indefinitely.
