Conceptify internals

How an artifact reaches your eyes

Every explanation Conceptify shows you is a single HTML file sitting in central storage — plain enough to open in any browser, forever. Yet inside the app that same file runs inside a double-locked box: a custom URL scheme on the outside, an origin-less iframe on the inside. This artifact walks the path a file takes from disk to pixels, and shows why the box is shaped the way it is.

The mental model

The viewer never loads artifact files directly. Instead the Rust core registers a custom URL scheme — artifact:// — and answers each request itself: read the bytes from central storage, attach security headers, inject the bridge script that carries comment clicks back to the shell, and hand the result to WebKit. Because artifact:// is a different scheme from the app's own UI, WebKit treats artifact content as a foreign website — real origin isolation, enforced by the browser engine rather than by convention.

The second lock is the iframe itself. It is sandboxed without allow-same-origin, which places the document in an opaque origin: an origin equal to nothing, not even itself.

Definition — opaque origin

An origin that compares unequal to every origin, including its own. Documents in an opaque origin get no storage (localStorage and friends throw), no cookies, and no way to reach the embedding page — window.parent is cross-origin territory.

What happens on every load

  1. Resolve. The viewer asks for artifact://thread/<slug>/artifact.v3.html; the protocol handler maps the URL to a file under central storage — no path from the request ever touches the filesystem directly.
  2. Read. The bytes are read from disk exactly as the generating agent saved them. Versions are immutable files, so a crash mid-read can never observe a half-written artifact.
  3. Fortify. The response gains its Content-Security-Policy header and the sandbox attributes are already waiting on the iframe — two locks, applied in two different places.
  4. Bridge. A small script is injected at serve time (never written to disk) that reports text selections and data-cfy-id clicks to the shell for commenting.
  5. Render. WebKit parses the document in its opaque origin. Everything the artifact needs — styles, diagrams, data — is already inline, so first paint is also final paint.

Insight

Because the policy travels on the response rather than in the file, tightening security ships with an app update and applies retroactively to every artifact ever saved. The files on disk never need to be edited — which is exactly what you want from documents meant to outlive the app that made them.

The policy, line by line

The reference policy is small enough to read in one breath. Everything is denied by default; inline code and styles are allowed (the artifact is self-contained, so "inline" is everything); and exactly one CDN host is carved out for the opt-in interactive tier.

Warning

connect-src 'none' means no network I/O of any kind from artifact JavaScript — fetch, XMLHttpRequest, and WebSockets all fail, even toward the allowlisted CDN. Any data an interactive artifact wants to chart, animate, or filter must already be embedded in the file.

Why the opaque origin matters

The custom scheme keeps artifact content out of the app's origin; the opaque origin keeps artifacts out of each other's. Two artifacts loaded over the same scheme would otherwise share an origin — and with it, storage and the ability to script one another across history navigation. Opaque origins make every load an island.

Deep dive: what allow-same-origin would give away

Adding allow-same-origin to the sandbox would hand the document a real origin — artifact://'s own. From there, every artifact could read and write a shared localStorage, plant state for the next artifact to find, and — since the bridge script runs in the same document — inspect whatever the bridge can see.

The sandbox flag you leave out is worth more than the ten you put in.

With the flag omitted, none of that surface exists. The bridge still works because it talks outward via postMessage, which functions fine from an opaque origin; the shell verifies the message source rather than trusting the origin string.

One file, two renderers

The same bytes render in the app's viewer and in any browser via "Open in browser" — that is the portability guarantee. What changes is not the document but the room it finds itself in:

Capability In-app viewer Standalone browser
Network access Blocked — connect-src 'none' Unrestricted (artifact still works offline by design)
Storage APIs Throw — opaque origin has none Available, but never relied upon
Comment anchors Live — bridge reports clicks on data-cfy-id Inert attributes, invisible to the reader
External links Inert (no navigation permission) Open normally
Tier-2 CDN libraries Load from the one allowlisted host Load normally; styled fallback when offline
The document is identical in both rooms; only the walls differ.

Threat model note

All of this is containment and hygiene, not adversarial hardening: artifacts come from the user's own agent, not from strangers. The box exists so that a buggy artifact — or a clever one — degrades into a boring document instead of a problem.

The one sanctioned sidecar

Everything an artifact needs lives inline — with a single carve-out. Explainer video clips are too heavy to embed, so they are uploaded to central storage before the artifact that references them (conceptify save-asset) and fetched at view time through a second custom scheme, cfy-asset://, whose handler answers byte-range requests so the player can seek. The takeaway

The whole design reduces to one sentence: the file is permanent, the walls are per-response. In practice that means:

  • Artifacts stay plain, portable HTML — openable anywhere, forever, offline.
  • Isolation comes from the serving path (scheme + CSP + sandbox), so it can be tightened without touching a single saved file.
  • The bridge is injected at serve time and never exists on disk, so the browser-opened copy is exactly the document and nothing more.
